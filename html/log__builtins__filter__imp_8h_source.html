180 /**
181  This is part of the 5.7 emulation:
182  If --log_error_verbosity is changed, we generate an
183  artificial filter rule from it here.
184 
185  For this filtering to be active, @@global.log_error_services
186  has to feature "log_filter_internal", as it does by default.
187  When that is the case, one or both of log_error_verbosity and
188  log_error_suppression_list (see below) may be used.
189  Only one of "log_filter_internal" and "log_filter_dragnet"
190  should be used at a time.
191 
192  @param verbosity log_error_verbosity style, range(1,3)
193  1:errors, 2:+=warnings, 3:+=notes
194 
195  @retval 0: success
196  @retval !0: failure
197 */
198 int log_builtins_filter_update_verbosity(int verbosity);
199 
200 /**
201  @@global.log_error_suppression_list accepts a comma-separated
202  list of error-codes that should not be included in the error-log.
203  Events with a severity of System or Error can not be filtered
204  in this way and will always be forwarded to the log-sinks.
205 
206  This provides simple filtering for cases where the flexibility
207  of the loadable filter-language is not needed. (The same engine
208  is used however, just with a more limited interface.)
209 
210  For this filtering to be active, @@global.log_error_services has
211  to feature "log_filter_internal", as it does by default. When that
212  is the case, one or both of log_error_verbosity and this variable
213  may be used. Only one of "log_filter_internal" and "log_filter_dragnet"
214  should be used at a time.
215 
216  The semantics follow that of our system variables; that is to say,
217  when called with update==false, the function acts as a check-function
218  that validates the entire list given to it; when called with
219  update==true, it creates filter-rules from the list items. This way,
220  we either create all rules, or no rules, rather than ending up with
221  an incomplete rule-set when we encounter a problem in the input.
222 
223  The return value encodes the location in the argument where the
224  failure occurred, like so:
225  - if 0 is returned, no issues were detected
226  - if a value less than zero is returned, -(retval + 1) is the
227  byte position (counting from 0) in the list argument at
228  which point the failure was detected
229 
230  @param list list of error-codes that should not appear
231  in the error-log
232  @param update false: verify list only
233  true: create filtering rules from suppression list
234 
235  @retval 0: success
236  @retval !0: failure (see above)
237 */
238 int log_builtins_filter_parse_suppression_list(char *list, bool update);
