97 /**
98  The structure Sort_addon_field describes the layout
99  for field values appended to sorted values in records to be sorted
100  in the sort buffer.
101 
102  Null bit maps for the appended values is placed before the values
103  themselves. Offsets are from the last sorted field.
104 
105  The structure is used to store values of the additional fields
106  in the sort buffer. It is used also when these values are read
107  from a temporary file/buffer in 'Filesort_info::unpack_addon_fields'.
108 */
109 
110 struct Sort_addon_field { /* Sort addon packed field */
111  Field *field; /* Original field */
112  uint null_offset; /* Offset to to null bit from the last sorted field */
113  uint max_length; /* Maximum length in the sort buffer */
114  uint8 null_bit; /* Null bit mask for the field */
115 };

209 /**
210  There are several record formats for sorting:
211 @verbatim
212  |<key a><key b>... | ( <null row flag> | <rowid> | ) * num_tables
213  / m_fixed_sort_length / ( 0 or 1 bytes | ref_len / )
214 @endverbatim
215 
216  or with "addon fields"
217 @verbatim
218  |<key a><key b>... |<null bits>|<field a><field b>...|
219  / m_fixed_sort_length / addon_length /
220 @endverbatim
221 
222  The packed format for "addon fields"
223 @verbatim
224  |<key a><key b>... |<length>|<null bits>|<field a><field b>...|
225  / m_fixed_sort_length / addon_length /
226 @endverbatim
227 
228  For packed addon fields, fields are not stored if the table is nullable and
229  has its NULL bit set.
230 
231  All the figures above are depicted for the case of fixed-size keys, with
232  appropriate padding. Fixed-size keys can be compared/sorted using memcmp().
233 
234  The packed (variable length) format for keys:
235 @verbatim
236  |<keylen>|<varkey a><key b>...<hash>|<(null_row,rowid) * num_tables> or <addons> |
237  / 4 bytes/ keylen bytes / (0/1 + ref_len) * num_tables or addon_length /
238 @endverbatim
239 
240  Variable-size keys must be compared piece-by-piece, using type information
241  about each individual key part, @see cmp_varlen_keys.
242 
243  All the record formats consist of a (possibly composite) key,
244  followed by a (possibly composite) payload.
245  The key is used for sorting data. Once sorting is done, the payload is
246  stored in some buffer, and read by some RowIterator.
247 
248 <dl>
249 <dt>@<key@>
250  <dd> Fields are fixed-size, specially encoded with
251  Field::make_sort_key() so we can do byte-by-byte compare.
252 <dt>@<length@>
253  <dd> Contains the *actual* packed length (after packing) of
254  everything after the sort keys.
255  The size of the length field is 2 bytes,
256  which should cover most use cases: addon data <= 65535 bytes.
257  This is the same as max record size in MySQL.
258 <dt>@<null bits@>
259  <dd> One bit for each nullable table and field, indicating whether
260  the table/field is NULL or not. May have size zero if no fields
261  or rows are nullable. NULL bits for rows (on nullable tables),
262  if any, always come before NULL bits for fields.
263 
264 <dt>@<field xx@>
265  <dd> Are stored with field->pack(), and retrieved with
266  field->unpack().
267  Addon fields within a record are stored consecutively, with no
268  "holes" or padding. They will have zero size for NULL values.
269 <dt>@<keylen@>
270  <dd> Contains the *actual* packed length of all the keys.
271  We may have an arbitrary mix of fixed and variable-sized keys.
272 <dt>@<hash@>
273  <dd> Optional 8 byte hash, used for GROUPing of JSON values.
274 <dt>@<varkey@>
275  <dd> Used for JSON and variable-length string values, the format is:
276 </dl>
277 @verbatim
278  |<null value>|<key length>|<sort key> |
279  / 1 byte / 4 bytes / key length bytes /
280 @endverbatim
281 <dl>
282 <dt>@<null value@>
283  <dd> 0x00 for NULL. 0xff for NULL under DESC sort. 0x01 for NOT NULL.
284 <dt>@<key length@>
285  <dd> The length of the sort key, *including* the four bytes for the
286  key length. Does not exist if the field is NULL.
287 </dl>
288  */
