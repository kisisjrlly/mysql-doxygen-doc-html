48  This class determines the optimal join order for tables within

50  with semi-joined tables from embedded subqueries.
51

182 /**
183  Calculate 'Post read filtering' effect of JOIN::conds for table
184  'tab'. Only conditions that are not directly involved in the chosen
185  access method shall be included in the calculation of this 'Post
186  read filtering' effect.
187 
188  The function first identifies fields that are directly used by the
189  access method. This includes columns used by range and ref access types,
190  and predicates on the identified columns (if any) will not be taken into
191  account when the filtering effect is calculated.
192 
193  The function will then calculate the filtering effect of any predicate
194  that applies to 'tab' and is not depending on the columns used by the
195  access method. The source of information with highest accuracy is
196  always preferred and is as follows:
197  1) Row estimates from the range optimizer
198  2) Row estimates from index statistics (records per key)
199  3) Guesstimates
200 
201  Thus, after identifying columns that are used by the access method,
202  the function will look for rows estimates made by the range optimizer.
203  If found, the estimates from the range optimizer are calculated into
204  the filtering effect.
205 
206  The function then goes through JOIN::conds to get estimates from any
207  remaining predicate that applies to 'tab' and does not depend on any
208  tables that come later in the join sequence. Predicates that depend on
209  columns that are either used by the access method or used in the row
210  estimate from the range optimizer will not be considered in this phase.
211 
212  @param tab The table condition filtering effect is calculated
213  for
214  @param keyuse Describes the 'ref' access method (if any) that is
215  chosen
216  @param used_tables Tables earlier in the join sequence than 'tab'
217  @param fanout The number of rows read by the chosen access
218  method for each row combination of previous tables
219  @param is_join_buffering Whether or not condition filtering is about
220  to be calculated for an access method using join
221  buffering.
222  @param write_to_trace Wheter we should print the filtering effect calculated
223  by histogram statistics and the final aggregated filtering
224  effect to optimizer trace.
225  @param parent_trace The parent trace object where the final aggregated
226  filtering effect will be printed if "write_to_trace" is
227  set to true.
228 
229  @return the 'post read filtering' effect (between 0 and 1) of
230  JOIN::conds
231 */
232 float calculate_condition_filter(const JOIN_TAB *const tab,
233  const Key_use *const keyuse,
234  table_map used_tables, double fanout,
235  bool is_join_buffering, bool write_to_trace,
236  Opt_trace_object &parent_trace);

238 class Join_tab_compare_default {
239  public:
240  /**
241  "Less than" comparison function object used to compare two JOIN_TAB
242  objects based on a number of factors in this order:
243 
244  - table before another table that depends on it (straight join,
245  outer join etc), then
246  - table before another table that depends on it to use a key
247  as access method, then
248  - table with smallest number of records first, then
249  - the table with lowest-value pointer (i.e., the one located
250  in the lowest memory address) first.
251 
252  @param jt1 first JOIN_TAB object
253  @param jt2 second JOIN_TAB object
254 
255  @note The order relation implemented by Join_tab_compare_default is not
256  transitive, i.e. it is possible to choose a, b and c such that
257  (a @< b) && (b @< c) but (c @< a). This is the case in the
258  following example:
259 
260  a: dependent = @<none@> found_records = 3
261  b: dependent = @<none@> found_records = 4
262  c: dependent = b found_records = 2
263 
264  a @< b: because a has fewer records
265  b @< c: because c depends on b (e.g outer join dependency)
266  c @< a: because c has fewer records
267 
268  This implies that the result of a sort using the relation
269  implemented by Join_tab_compare_default () depends on the order in
270  which elements are compared, i.e. the result is
271  implementation-specific.
272 
273  @return
274  true if jt1 is smaller than jt2, false otherwise
275  */
276  bool operator()(const JOIN_TAB *jt1, const JOIN_TAB *jt2) const;
277 };
