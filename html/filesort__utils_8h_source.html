42 /**
43  Buffer used for storing records to be sorted. The records are stored in
44  a series of buffers that are allocated incrementally, growing 50% each
45  time, similar to how a MEM_ROOT works. This allows the user to set a large
46  maximum buffer size without getting huge allocations for sorting small result
47  sets. It means that if you actually _do_ use the entire buffer, there will be
48  more allocations than one large allocation up-front, but this is a worthwhile
49  tradeoff (those allocation will tend to disappear into the cost of actually
50  getting all the rows and sorting them).
51 
52  In addition, Filesort_buffer stores a vector of pointers to the beginning of
53  each record. It is these pointers that are actually sorted in filesort.
54  If the records are small, this can add up to overhead on top of the amount of
55  memory the user expected to use. We _do_ take already allocated pointers into
56  account when calculating how big a new block can be, so the amount of badness
57  is bounded:
58 
59  Assume that we have set maximum record size to infinity, but that in
60  practice, they are are about the smallest size possible (4-byte sort key plus
61  4-byte rowid) and that we are on a 64-bit system. Then, the worst possible
62  overhead is that we use as much space on pointers as the size of the last
63  (largest) block. We can look at the two possible extremes:
64 
65  - Smallest possible sort buffer (32 kB): 32 kB overhead.
66  - A huge sort buffer (x kB): If the last block is y kB, the total size will
67  be y + 2/3y + (2/3)Â²y + ... = 3y, which means the last block is 1/3 of the
68  total size. Thus, pointer overhead will be no worse than 33%.
69 
70  In most practical cases, it will be much better than this. In particular,
71  even when allocating a block (where we don't yet know how many records will
72  fit), we allow space for the record pointers we'd need given maximum-sized
73  rows.
74 
75  The buffer must be kept available for multiple executions of the
76  same sort operation, so one can call reset() for reuse. Again similar
77  to MEM_ROOT, this keeps the last (largest) block and discards the others.
78 */

99  /**
100  Where should the next record be stored?
101 
102  If a block is returned, it is always at least "min_size" bytes long.
103  If the returned block is not large enough for your purposes,
104  call get_next_record_pointer() again with a larger value of min_size than
105  the size you got back. Just increasing the size by one byte is fine;
106  the class will still try to make exponentially larger blocks each time.
107 
108  If there's no room for a record of the given size, returns nullptr.
109 
110  After you've written data to the given record, call commit_used_memory()
111  with the number of bytes you've actually written. This ensures it will
112  not get reused for subsequent records.
113  */
114  Bounds_checked_array<uchar> get_next_record_pointer(size_t min_size) {
115  DBUG_ASSERT(min_size != 0xFFFFFFFFu);
116  // See if we need to allocate a new block.
117  if (m_next_rec_ptr == nullptr ||
118  m_next_rec_ptr + min_size > m_current_block_end) {
119  if (allocate_block(min_size)) return Bounds_checked_array<uchar>();
120  }
121 
122  // Allocate space within the current block.
123  return Bounds_checked_array<uchar>(m_next_rec_ptr,
124  m_current_block_end - m_next_rec_ptr);
125  }
126 
127  void commit_used_memory(size_t num_bytes) {
128  m_record_pointers.push_back(m_next_rec_ptr);
129  m_next_rec_ptr += num_bytes;
130  }

203  /**
204  Clears all rows, then returns a contiguous buffer of maximum size.
205  (This may or may not involve allocation.) This is for reusing the memory
206  for merge buffers, which requires the memory to be a single contiguous
207  chunk; one could in theory adjust merging to allow using multiple buffers
208  like sorting does, but once we need to merge, that means we've hit disk
209  anyway (or at the very least, need to talk to the OS' buffer cache),
210  and the cost of a single allocation is small compared to I/O.
211 
212  If you use this memory area, you cannot also use the Filesort_buffer to
213  store sort records (get_next_record_pointer etc.); that would use the
214  same memory.
215 
216  Can return nullptr, if allocation fails.
217  */
