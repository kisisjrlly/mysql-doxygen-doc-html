26 /**
27  @file composite_iterators.h
28 
29  A composite row iterator is one that takes in one or more existing iterators
30  and processes their rows in some interesting way. They are usually not bound
31  to a single table or similar, but are the inner (non-leaf) nodes of the
32  iterator execution tree. They consistently own their source iterator, although
33  not its memory (since we never allocate row iterators on the heap--usually on
34  a MEM_ROOT>). This means that in the end, you'll end up with a single root
35  iterator which then owns everything else recursively.
36 
37  SortingIterator is also a composite iterator, but is defined in its own file.
38  */

164 /**
165  Handles aggregation (typically used for GROUP BY) for the case where the rows
166  are already properly grouped coming in, ie., all rows that are supposed to be
167  part of the same group are adjacent in the input stream. (This could be
168  because they were sorted earlier, because we are scanning an index that
169  already gives us the rows in a group-compatible order, or because there is no
170  grouping.)
171 
172  AggregateIterator is special in that it's one of the very few row iterators
173  that actually change the shape of the rows; some columns are dropped as part
174  of aggregation, others (the aggregates) are added. For this reason (and also
175  because we need to make copies of the group expressions -- see Read()), it
176  conceptually always outputs to a temporary table. If we _are_ outputting to a
177  temporary table, that's not a problem -- we take over responsibility for
178  copying the group expressions from MaterializeIterator, which would otherwise
179  do it.
180 
181  However, if we are outputting directly to the user, we need somewhere to store
182  the output. This is solved by abusing the slice system; since we only need to
183  buffer a single row, we can set up just enough items in the
184  REF_SLICE_ORDERED_GROUP_BY slice, so that it can hold a single row. This row
185  is then used for our output, and we then switch to it just before the end of
186  Read() so that anyone reading from the buffers will get that output.
187  The caller knows the context about where our output goes, and thus also picks
188  the appropriate output slice for us.
189 
190  This isn't very pretty. What should be done is probably a more abstract
191  concept of sending a row around and taking copies of it if needed, as opposed
192  to it implicitly staying in the table's buffer. (This would also solve some
193  issues in EQRefIterator and when synthesizing NULL rows for outer joins.)
194  However, that's a large refactoring.
195  */

336  iterator many times. It is currently the only form of join we have.
337

451 /**
452  Handles materialization; the first call to Init() will scan the given iterator
453  to the end, store the results in a temporary table (optionally with
454  deduplication), and then Read() will allow you to read that table repeatedly
455  without the cost of executing the given subquery many times (unless you ask
456  for rematerialization).
457 
458  When materializing, MaterializeIterator takes care of evaluating any items
459  that need so, and storing the results in the fields of the outgoing table --
460  which items is governed by the temporary table parameters.
461 
462  Conceptually (although not performance-wise!), the MaterializeIterator is a
463  no-op if you don't ask for deduplication, and in some cases (e.g. when
464  scanning a table only once), we elide it. However, it's not necessarily
465  straightforward to do so by just not inserting the iterator, as the optimizer
466  will have set up everything (e.g., read sets, or what table upstream items
467  will read from) assuming the materialization will happen, so the realistic
468  option is setting up everything as if materialization would happen but not
469  actually write to the table; see StreamingIterator for details.
470 
471  MaterializeIterator conceptually materializes iterators, not JOINs or
472  SELECT_LEX_UNITs. However, there are many details that leak out
473  (e.g., setting performance schema batch mode, slices, reusing CTEs,
474  etc.), so we need to send them in anyway.
475  */

477  public:
