356 /// Maps compare function to strict weak ordering required by Priority_queue.
357 struct Key_rec_less {

359 
360  explicit Key_rec_less(KEY **keys)
361  : m_keys(keys), m_fun(key_rec_cmp), m_max_at_top(false) {}
362 
363  bool operator()(uchar *first, uchar *second) {
364  const int cmpval =

367  }

501  /**
502  MODULE full table scan
503 
504  This module is used for the most basic access method for any table
505  handler. This is to fetch all data through a full table scan. No
506  indexes are needed to implement this part.
507  It contains one method to start the scan (rnd_init) that can also be
508  called multiple times (typical in a nested loop join). Then proceeding
509  to the next record (rnd_next) and closing the scan (rnd_end).
510  To remember a record for later access there is a method (position)
511  and there is a method used to retrieve the record based on the stored
512  position.
513  The position can be a file position, a primary key, a ROWID dependent
514  on the handler below.
515

517  without rnd_end() in between (it only makes sense if scan=1).
518  then the second call should prepare for the new table scan
519  (e.g if rnd_init allocates the cursor, second call should
520  position it to the start of the table, no need to deallocate
521  and allocate it again.
522  @see handler.h
523  @{
524  */

533  /**
534  MODULE index scan
535 
536  This part of the handler interface is used to perform access through
537  indexes. The interface is defined as a scan interface but the handler
538  can also use key lookup if the index is a unique index or a primary
539  key index.
540  Index scans are mostly useful for SELECT queries but are an important
541  part also of UPDATE, DELETE, REPLACE and CREATE TABLE table AS SELECT
542  and so forth.
543  Naturally an index is needed for an index scan and indexes can either
544  be ordered, hash based. Some ordered indexes can return data in order
545  but not necessarily all of them.
546  There are many flags that define the behavior of indexes in the
547  various handlers. These methods are found in the optimizer module.
548  -------------------------------------------------------------------------
549 
550  index_read is called to start a scan of an index. The find_flag defines
551  the semantics of the scan. These flags are defined in
552  include/my_base.h
553  index_read_idx is the same but also initializes index before calling doing
554  the same thing as index_read. Thus it is similar to index_init followed
555  by index_read. This is also how we implement it.
556 
557  index_read/index_read_idx does also return the first row. Thus for
558  key lookups, the index_read will be the only call to the handler in
559  the index scan.
560

562  any end processing needed.
