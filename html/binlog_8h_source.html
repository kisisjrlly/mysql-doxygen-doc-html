552  /**
553  Flush and commit the transaction.
554 
555  This will execute an ordered flush and commit of all outstanding
556  transactions and is the main function for the binary log group
557  commit logic. The function performs the ordered commit in four stages.
558 
559  Pre-condition: transactions should have called ha_prepare_low, using
560  HA_IGNORE_DURABILITY, before entering here.
561 
562  Stage#0 implements slave-preserve-commit-order for applier threads that
563  write the binary log. i.e. it forces threads to enter the queue in the
564  correct commit order.
565 
566  The stage#1 flushes the caches to the binary log and under
567  LOCK_log and marks all threads that were flushed as not pending.
568 
569  The stage#2 syncs the binary log for all transactions in the group.
570 
571  The stage#3 executes under LOCK_commit and commits all transactions in
572  order.
573 
574  There are three queues of THD objects: one for each stage.
575  The Commit_order_manager maintains it own queue and its own order for the
576  commit. So Stage#0 doesn't maintain separate StageID.
577 
578  When a transaction enters a stage, it adds itself to a queue. If the queue
579  was empty so that this becomes the first transaction in the queue, the
580  thread is the *leader* of the queue. Otherwise it is a *follower*. The
581  leader will do all work for all threads in the queue, and the followers
582  will wait until the last stage is finished.
583 
584  Stage 0 (SLAVE COMMIT ORDER):
585  1. If slave-preserve-commit-order and is slave applier worker thread, then
586  waits until its turn to commit i.e. till it is on the top of the queue.
587  2. When it reaches top of the queue, it signals next worker in the commit
588  order queue to awake.
589 
590  Stage 1 (FLUSH):
591  1. Sync the engines (ha_flush_logs), since they prepared using non-durable
592  settings (HA_IGNORE_DURABILITY).
593  2. Generate GTIDs for all transactions in the queue.
594  3. Write the session caches for all transactions in the queue to the binary
595  log.
596  4. Increment the counter of prepared XIDs.
597 
598  Stage 2 (SYNC):
599  1. If it is time to sync, based on the sync_binlog option, sync the binlog.
600  2. If sync_binlog==1, signal dump threads that they can read up to the
601  position after the last transaction in the queue
602 
603  Stage 3 (COMMIT):
604  This is performed by each thread separately, if binlog_order_commits=0.
605  Otherwise by the leader does it for all threads.
606  1. Call the after_sync hook.
607  2. update the max_committed counter in the dependency_tracker
608  3. call ha_commit_low
609  4. Call the after_commit hook
610  5. Update gtids
611  6. Decrement the counter of prepared transactions
612 
613  If the binary log needs to be rotated, it is done after this. During
614  rotation, it takes a lock that prevents new commit groups from executing the
615  flush stage, and waits until the counter of prepared transactions becomes 0,
616  before it creates the new file.
617

619  @param[in] all This is @c true if this is a real transaction commit, and
620  @c false otherwise.
