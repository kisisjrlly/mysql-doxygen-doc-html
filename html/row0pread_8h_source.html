56 /** The core idea is to find the left and right paths down the B+Tree.These
57 paths correspond to the scan start and scan end search. Follow the links
58 at the appropriate btree level from the left to right and split the scan
59 on each of these sub-tree root nodes.
60 
61 If the user has set the maximum number of threads to use at say 4 threads
62 and there are 5 sub-trees at the selected level then we will split the 5th
63 sub-tree dynamically when it is ready for scan.
64 
65 We want to allow multiple parallel range scans on diffrent indexes at the
66 same time. To achieve this split out the scan context (Scan_ctx) from the
67 execution context (Ctx). The Scan_ctx has the index and transaction
68 information and the Ctx keeps track of the cursor for a specific thread
69 during the scan.
70 
71 To start a scan we need to instantiate a Parallel_reader. A parallel reader
72 can contain several Scan_ctx instances and a Scan_ctx can contain several
73 Ctx instances. Its' the Ctx instances that are eventually executed.
74 
75 This design allows for a single Parallel_reader to scan multiple indexes
76 at once. Each index range scan has to be added via its add_scan() method.
77 This functionality is required to handle parallel partition scans because
78 partitions are separate indexes. This can be used to scan completely
79 different indexes and tables by one instance of a Parallel_reader.
80 
81 To solve the imbalance problem we dynamically split the sub-trees as and
82 when required. e.g., If you have 5 sub-trees to scan and 4 threads then
83 it will tag the 5th sub-tree as "to_be_split" during phase I (add_scan()),
84 the first thread that finishes scanning the first set of 4 partitions will
85 then dynamically split the 5th sub-tree and add the newly created sub-trees
86 to the execution context (Ctx) run queue in the Parallel_reader. As the
87 other threads complete their sub-tree scans they will pick up more execution
88 contexts (Ctx) from the Parallel_reader run queue and start scanning the
89 sub-partitions as normal.
90 
91 Note: The Ctx instances are in a virtual list. Each Ctx instance has a
92 range to scan. The start point of this range instance is the end point
93 of the Ctx instance scanning values less than its start point. A Ctx
94 will scan from [Start, End) rows. We use std::shared_ptr to manage the
95 reference counting, this allows us to dispose of the Ctx instances
96 without worrying about dangling pointers.
97 
98 NOTE: Secondary index scans are not supported currently. */
