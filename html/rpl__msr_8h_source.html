55 /**
56  Class to store all the Master_info objects of a slave
57  to access them in the replication code base or performance
58  schema replication tables.
59 
60  In a Multisourced replication setup, a slave connects
61  to several masters (also called as sources). This class
62  stores the Master_infos where each Master_info belongs
63  to a slave.
64 
65  The important objects for a slave are the following:
66  i) Master_info and Relay_log_info (slave_parallel_workers == 0)
67  ii) Master_info, Relay_log_info and Slave_worker(slave_parallel_workers >0 )
68 
69  Master_info is always assosiated with a Relay_log_info per channel.
70  So, it is enough to store Master_infos and call the corresponding
71  Relay_log_info by mi->rli;
72 
73  This class is not yet thread safe. Any part of replication code that
74  calls this class member function should always lock the channel_map.
75 
76  Only a single global object for a server instance should be created.
77 
78  The two important data structures in this class are
79  i) C++ std map to store the Master_info pointers with channel name as a key.
80  These are the base channel maps.
81  @todo Convert to boost after it's introduction.
82 
83  ii) C++ std map to store the channel maps with a channel type as its key.
84  This map stores slave channel maps, group replication channels or others
85  iii) An array of Master_info pointers to access from performance schema
86  tables. This array is specifically implemented in a way to make
87  a) pfs indices simple i.e a simple integer counter
88  b) To avoid recalibration of data structure if master info is deleted.
89  * Consider the following high level implementation of a pfs table
90  to make a row.
91  @code
92  highlevel_pfs_funciton()
93  {
94  while(replication_table_xxxx.rnd_next())
95  {
96  do stuff;
97  }
98  }
99  @endcode
100  However, we lock channel_map lock for every rnd_next(); There is a gap
101  where an addition/deletion of a channel would rearrange the map
102  making the integer indices of the pfs table point to a wrong value.
103  Either missing a row or duplicating a row.
104 
105  We solve this problem, by using an array exclusively to use in
106  replciation pfs tables, by marking a master_info defeated as 0
107  (i.e NULL). A new master info is added to this array at the
108  first NULL always.
109 */

254  return 0;
255  else

388 /**
389  The class is a container for all the per-channel filters, both a map of
390  Rpl_filter objects and a list of Rpl_pfs_filter objects.
391  It maintains a filter map which maps a replication filter to a channel
392  name. Which is needed, because replication channels are not created and
393  channel_map is not filled in when these global and per-channel replication
394  filters are evaluated with current code frame.
395  In theory, after instantiating all channels from the repository and throwing
396  all the warnings about the filters configured for non-existent channels, we
397  can forget about its global object rpl_channel_filters and rely only on the
398  global and per channel Rpl_filter objects. But to avoid holding the
399  channel_map.rdlock() when quering P_S.replication_applier_filters table,
400  we keep the rpl_channel_filters. So that we just need to hold the small
401  rpl_channel_filters.rdlock() when quering P_S.replication_applier_filters
402  table. Many operations (RESET SLAVE [FOR CHANNEL], START SLAVE, INIT SLAVE,
403  END SLAVE, CHANGE MASTER TO, FLUSH RELAY LOGS, START CHANNEL, PURGE CHANNEL,
404  and so on) hold the channel_map.wrlock().
405 
406  There is one instance, rpl_channel_filters, created globally for Multisource
407  channel filters. The rpl_channel_filters is created when the server is
408  started, destroyed when the server is stopped.
409 */
410 class Rpl_channel_filters {
411  private:
412  /* Store all replication filters with channel names. */
413  filter_map channel_to_filter;
414  /* Store all Rpl_pfs_filter objects in the channel_to_filter. */
415  std::vector<Rpl_pfs_filter> rpl_pfs_filter_vec;
416  /*
417  This lock was designed to protect the channel_to_filter from reading,
418  adding, or removing its objects from the map. It is used to preventing
419  the following commands to run in parallel:
420  RESET SLAVE ALL [FOR CHANNEL '<channel_name>']
421  CHANGE MASTER TO ... FOR CHANNEL
422  SELECT FROM performance_schema.replication_applier_filters
423 
424  Please acquire a wrlock when modifying the map structure (RESET SLAVE ALL
425  [FOR CHANNEL '<channel_name>'], CHANGE MASTER TO ... FOR CHANNEL).
426  Please acqurie a rdlock when querying existing filter(s) (SELECT FROM
427  performance_schema.replication_applier_filters).
428 
429  Note: To modify the object from the map, please see the protection of
430  m_rpl_filter_lock in Rpl_filter.
431  */
432  Checkable_rwlock *m_channel_to_filter_lock;
433
