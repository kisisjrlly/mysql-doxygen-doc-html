26 /// @file
27 ///
28 /// This file contains the HashJoinRowBuffer class and related
29 /// functions/classes.
30 ///
31 /// A HashJoinBuffer is a row buffer that can hold a certain amount of rows.
32 /// The rows are stored in a hash table, which allows for constant-time lookup.
33 /// The HashJoinBuffer maintains its own internal MEM_ROOT, where all of the
34 /// data is allocated.
35 ///
36 /// The HashJoinBuffer contains an operand with rows from one or more tables,
37 /// keyed on the value we join on. Consider the following trivial example:
38 ///
39 /// SELECT t1.data FROM t1 JOIN t2 ON (t1.key = t2.key);
40 ///
41 /// Let us say that the table "t2" is stored in a HashJoinBuffer. In this case,
42 /// the hash table key will be the value found in "t2.key", since that is the
43 /// join condition that belongs to t2. If we have multiple equalities, they
44 /// will be concatenated together in order to form the hash table key. The hash
45 /// table key is bascially an std::string_view, and should be replaced when we
46 /// go to C++17.
47 ///
48 /// In order to store a row, we use the function StoreFromTableBuffers. See the
49 /// comments attached to the function for more details.
50 ///
51 /// The amount of memory a HashJoinBuffer instance can use is limited by the
52 /// system variable "join_buffer_size". However, note that we check whether we
53 /// have exceeded the memory limit _after_ we have inserted data into the row
54 /// buffer. As such, we will probably use a little bit more memory than
55 /// specified by join_buffer_size.
56 ///
57 /// Some basic profiling shows that the majority of the time is used on
58 /// constructing the hash table. It would be a future improvement to replace
59 /// std::unordered_multimap with a more performant hash table, as the literature
60 /// suggests that there are better alternatives (see e.g.
61 /// https://probablydance.com/2017/02/26/i-wrote-the-fastest-hashtable/).
62 ///
63 /// The primary use case for these classes is, as the name implies,
64 /// for implementing hash join.

172 /// The key type for the hash structure in HashJoinRowBuffer.
173 ///
174 /// A key consists of the value from one or more columns, taken from the join
175 /// condition(s) in the query. E.g., if the join condition is
176 /// (t1.col1 = t2.col1 AND t1.col2 = t2.col2), the key is (col1, col2), with the
177 /// two key parts concatenated together.
178 ///
179 /// What the data actually contains depends on the comparison context for the
180 /// join condition. For instance, if the join condition is between a string
181 /// column and an integer column, the comparison will be done in a string
182 /// context, and thus the integers will be converted to strings before storing.
183 /// So the data we store in the key are in some cases converted, so that we can
184 /// hash and compare them byte-by-byte (i.e. decimals), while other types are
185 /// already comparable byte-by-byte (i.e. integers), and thus stored as-is.
186 ///
187 /// Note that the key data can come from items as well as fields if the join
188 /// condition is an expression. E.g. if the join condition is
189 /// UPPER(t1.col1) = UPPER(t2.col1), the join key data will come from an Item
190 /// instead of a Field.
191 ///
192 /// The Key class never takes ownership of the data. As such, the user must
193 /// ensure that the data has the proper lifetime. When storing rows in the row
194 /// buffer, the data must have the same lifetime as the row buffer itself.
195 /// When using the Key class for lookups in the row buffer, the same lifetime is
196 /// not needed; the key object is only needed when the lookup is done.
197 ///
198 /// When we move to C++17, this class should most likely be replaced with
199 /// std::string_view.
200 class Key {
201  public:
202  /// Note that data can be nullptr if size is 0.
203  Key(const uchar *data, size_t size) : m_data(data), m_size(size) {}
204 
205  bool operator==(const Key &other) const {
206  if (other.size() != size()) {
207  return false;
208  } else if (size() == 0) {
209  return true;
210  }
211 
212  return memcmp(other.data(), data(), size()) == 0;
213  }
214 
215  const uchar *data() const { return m_data; }
216 
217  size_t size() const { return m_size; }
218 
219  private:
220  const uchar *m_data;
221  const size_t m_size;
222 };

229 /// We rely on xxHash64 to do the hashing of the key. xxHash64 was chosen as it
230 /// both is very fast _and_ produces reasonably good-quality hashes
231 /// (see https://github.com/rurban/smhasher).
232 class KeyHasher {
233  public:
234  explicit KeyHasher(uint32_t seed) : m_seed(seed) {}
235 
236  size_t operator()(hash_join_buffer::Key key) const {
237  if (key.size() == 0) return kZeroKeyLengthHash;
238  return MY_XXH64(key.data(), key.size(), m_seed);
239  }
240 
241  private:
242  // An arbitrary hash value for the empty string, to avoid the hash function
243  // from doing arithmetic on nullptr, which is undefined behavior.
244  static constexpr size_t kZeroKeyLengthHash = 2669509769;
245  const uint32_t m_seed;
246 };
