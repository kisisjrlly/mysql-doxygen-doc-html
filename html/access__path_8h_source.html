74 /**
75  Access paths are a query planning structure that correspond 1:1 to iterators,
76  in that an access path contains pretty much exactly the information
77  needed to instantiate given iterator, plus some information that is only
78  needed during planning, such as costs. (The new join optimizer will extend
79  this somewhat in the future. Some iterators also need the query block,
80  ie., JOIN object, they are part of, but that is implicitly available when
81  constructing the tree.)
82 
83  AccessPath objects build on a variant, ie., they can hold an access path of
84  any type (table scan, filter, hash join, sort, etc.), although only one at the
85  same time. Currently, they contain 32 bytes of base information that is common
86  to any access path (type identifier, costs, etc.), and then up to 40 bytes
87  that is type-specific (e.g. for a table scan, the TABLE object). It would be
88  nice if we could squeeze it down to 64 and fit a cache line exactly, but it
89  does not seem to be easy without fairly large contortions.
90 
91  We could have solved this by inheritance, but the fixed-size design makes it
92  possible to replace an access path when a better one is found, without
93  introducing a new allocation, which will be important when using them as a
94  planning structure.
95  */
