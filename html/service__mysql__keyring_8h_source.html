30 /**
31  @ingroup group_ext_plugin_services
32 
33  This service allows plugins to interact with key store backends.
34 
35  A key currently is a blob of binary data, defined by a string
36  key type, that's meanigfull to the relevant backend.
37  Typical key_type values include "AES", "DES", "DSA" etc.
38  There's no length in the type, since it's defined by the number of bytes
39  the key takes.
40 
41  A key is uniquely identified by the key_id and the user_id values, i.e.
42  all keys are assigned to a particular user.
43  There's only one exception to that: a single category for instance keys
44  that are not associated with any particular user id. This is signified
45  to the APIs by supplying NULL for user_id.
46  Plugins would typically pass user accounts to the user_id parameter, or
47  NULL if there's no user account to associate the key with.
48 
49  Not all backends must implement all of the functions defined
50  in this interface.
51 
52  The plugin service is a "bridge service", i.e. facade with no
53  real functionality that just calls the actual keyring plugin APIs.
54  This is needed to allow other plugins to call into the keyring
55  plugins and thus overcome the limitation that only the server
56  can call plugins.
57 
58  @sa st_mysql_keyring
59 */

61  /**

64  */

66  const void *, size_t);
