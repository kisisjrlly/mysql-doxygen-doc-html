85 /**
86  Implementation of a dictionary client.
87 
88  The dictionary client provides a unified interface to accessing dictionary
89  objects. The client is a member of the THD, and is typically used in
90  server code to access the dictionary. When we refer to "the user" below,
91  we mean the server code using the dictionary client.
92 
93  The main task of the client is to access a shared cache to retrieve
94  dictionary objects. The shared cache, in its turn, will access the
95  dictionary tables if there is a cache miss.
96 
97  To support cache eviction, the shared cache must keep track of which
98  clients that have acquired an object. When a client acquires an object
99  from the shared cache for the first time, it is added to a client local
100  object registry. Further acquisition of the same object from the client
101  will get the object from the client's registry. Thus, the usage tracking
102  in the shared cache only keep track of the number of clients currently
103  using the object, and hence, there must be an operation that complements
104  acquisition, to inform the shared cache that the object is not used
105  anymore. This complementing operation is called releasing the object.
106 
107  To manage releasing objects, the Auto_releaser class provides some
108  support. When an auto releaser is instantiated, it will keep track of
109  the objects that are acquired from the shared cache in its lifetime.
110  Auto releasers may be nested or stacked, and the current releaser is
111  the one at the top of the stack. The auto releaser stack is associated
112  with a dictionary client instance. When the auto releaser goes out
113  of scope, it will release all objects that have been acquired from the
114  shared cache in its lifetime. Objects retrieved earlier than that will
115  be automatically released by a releaser further down the auto releaser
116  stack. For more coarse grained control, there is a release method that
117  will release all objects acquired by the client.
118 
119  In addition to the auto releasers, the client has an object registry.
120  The registry holds pointers to all currently acquired objects. Thus,
121  the object registry is the union of the registers in the stack of
122  auto releasers. The client's object registry is used for looking up
123  objects, while the registers in the auto releasers are used for
124  releasing objects.
125 
126  The client also has a second registery of objects with uncommitted changes.
127  These are objects acquired by acquire_for_modification() or registered
128  with register_uncommitted_object(). These objects are only present in
129  the local registry and not in the shared cache. Once registered, the
130  objects can also be retrieved with normal acquire(). This means that
131  a given client has a view which includes uncommitted changes made
132  using the same client, while other clients do not see these changes.
133 
134  @note We must handle situations where an object is actually acquired from
135  the shared cache, while the dynamic cast to a subtype fails. We use
136  the auto release mechanism to achieve that.
137 
138  @note When a dictionary client method returns true, indicating that an
139  error has occurred, the error has been reported, either by the
140  client itself, or by the dictionary subsystem.
141 */

148  /**
149  Class to help releasing and deleting objects.
150 
151  This class keeps a register of shared objects that are automatically
152  released when the instance goes out of scope. When a new instance
153  is created, the encompassing dictionary client's current auto releaser
154  is replaced by this one, keeping a link to the old one. When the
155  auto releaser is deleted, it links the old releaser back in as the
156  client's current releaser.
157 
158  Shared objects that are added to the auto releaser will be released when
159  the releaser is deleted. Only the dictionary client is allowed to add
160  objects to the auto releaser.
161 
162  The usage pattern is that objects that are retrieved from the shared
163  dictionary cache are added to the current auto releaser. Objects that
164  are retrieved from the client's local object register are not added to
165  the auto releaser. Thus, when the releaser is deleted, it releases all
166  objects that have been retrieved from the shared cache during the
167  lifetime of the releaser.
168 
169  Similarly the auto releaser maintains a list of objects created
170  by acquire_uncached(). These objects are owned by the Auto_releaser
171  and are deleted when the auto releaser goes out of scope.
172  */

365  template <typename T>
366  void auto_delete(T *object) {
367 #ifndef DBUG_OFF
368  // Make sure we do not sign up a shared object for auto delete.
369  Cache_element<typename T::Cache_partition> *element = nullptr;
370  m_registry_committed.get(
371  static_cast<const typename T::Cache_partition *>(object), &element);
372  DBUG_ASSERT(element == nullptr);
373 
374  // Make sure we do not sign up an uncommitted object for auto delete.
375  m_registry_uncommitted.get(
376  static_cast<const typename T::Cache_partition *>(object), &element);
377  DBUG_ASSERT(element == nullptr);
378 
379  // We must require a top level non-default releaser to ensure a
380  // predictable life span of the objects.
381  DBUG_ASSERT(m_current_releaser != &m_default_releaser);
382 #endif
383 
384  m_uncached_objects.push_back(object);
385  }
