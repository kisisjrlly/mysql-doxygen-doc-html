102 /**
103  A Key_use represents an equality predicate of the form (table.column = val),
104  where the column is indexed by @c keypart in @c key and @c val is either a
105  constant, a column from other table, or an expression over column(s) from
106  other table(s). If @c val is not a constant, then the Key_use specifies an
107  equi-join predicate, and @c table must be put in the join plan after all
108  tables in @c used_tables.
109 
110  At an abstract level a Key_use instance can be viewed as a directed arc
111  of an equi-join graph, where the arc originates from the table(s)
112  containing the column(s) that produce the values used for index lookup
113  into @c table, and the arc points into @c table.
114 
115  For instance, assuming there is only an index t3(c), the query
116 
117  @code
118  SELECT * FROM t1, t2, t3
119  WHERE t1.a = t3.c AND
120  t2.b = t3.c;
121  @endcode
122 
123  would generate two arcs (instances of Key_use)
124 
125  @code
126  t1-- a ->- c --.
127  |
128  V
129  t3
130  ^
131  |
132  t2-- b ->- c --'
133  @endcode
134 
135  If there were indexes t1(a), and t2(b), then the equi-join graph
136  would have two additional arcs "c->a" and "c->b" recording the fact
137  that it is possible to perform lookup in either direction.
138 
139  @code
140  t1-- a ->- c --. ,-- c -<- b --- t2
141  ^ | | ^
142  | | | |
143  `-- a -<- c - v v-- c ->- b ----'
144  t3
145  @endcode
146 
147  The query
148 
149  @code
150  SELECT * FROM t1, t2, t3 WHERE t1.a + t2.b = t3.c;
151  @endcode
152 
153  can be viewed as a graph with one "multi-source" arc:
154 
155  @code
156  t1-- a ---
157  |
158  >-- c --> t3
159  |
160  t2-- b ---
161  @endcode
162 
163  The graph of all equi-join conditions usable for index lookup is
164  stored as an ordered sequence of Key_use elements in
165  JOIN::keyuse_array. See sort_keyuse() for details on the
166  ordering. Each JOIN_TAB::keyuse points to the first array element
167  with the same table.
168 */

317 /**
318  A position of table within a join order. This structure is primarily used
319  as a part of @c join->positions and @c join->best_positions arrays.
320 
321  One POSITION element contains information about:
322  - Which table is accessed
323  - Which access method was chosen
324  = Its cost and \#of output records
325  - Semi-join strategy choice. Note that there are two different
326  representation formats:
327  1. The one used during join optimization
328  2. The one used at plan refinement/code generation stage.
329  We call fix_semijoin_strategies_for_picked_join_order() to switch
330  between #1 and #2. See that function's comment for more details.
331 
332  - Semi-join optimization state. When we're running join optimization,
333  we main a state for every semi-join strategy which are various
334  variables that tell us if/at which point we could consider applying the
335  strategy.
336  The variables are really a function of join prefix but they are too
337  expensive to re-caclulate for every join prefix we consider, so we
338  maintain current state in join->positions[\#tables_in_prefix]. See
339  advance_sj_state() for details.
340 
341  This class has to stay a POD, because it is memcpy'd in many places.
342 */

344 struct POSITION {
345  /**
346  The number of rows that will be fetched by the chosen access
347  method per each row combination of previous tables. That is:
348 
349  rows_fetched = selectivity(access_condition) * cardinality(table)
350 
351  where 'access_condition' is whatever condition was chosen for
352  index access, depending on the access method ('ref', 'range',
353  etc.)
354 
355  @note For index/table scans, rows_fetched may be less than
356  the number of rows in the table because the cost of evaluating
357  constant conditions is included in the scan cost, and the number
358  of rows produced by these scans is the estimated number of rows
359  that pass the constant conditions. @see
360  Optimize_table_order::calculate_scan_cost() . But this is only during
361  planning; make_join_readinfo() simplifies it for EXPLAIN.
362  */
363  double rows_fetched;
364 
365  /**
366  Cost of accessing the table in course of the entire complete join
367  execution, i.e. cost of one access method use (e.g. 'range' or
368  'ref' scan ) multiplied by estimated number of rows from tables
369  earlier in the join sequence.
370 
371  read_cost does NOT include cost of processing rows within the
372  executor (row_evaluate_cost).
373  */
374  double read_cost;
375 
376  /**
377  The fraction of the 'rows_fetched' rows that will pass the table
378  conditions that were NOT used by the access method. If, e.g.,
379 
380  "SELECT ... WHERE t1.colx = 4 and t1.coly @> 5"
381 
382  is resolved by ref access on t1.colx, filter_effect will be the
383  fraction of rows that will pass the "t1.coly @> 5" predicate. The
384  valid range is 0..1, where 0.0 means that no rows will pass the
385  table conditions and 1.0 means that all rows will pass.
386 
387  It is used to calculate how many row combinations will be joined
388  with the next table, @see prefix_rowcount below.
389 
390  @note With condition filtering enabled, it is possible to get
391  a fanout = rows_fetched * filter_effect that is less than 1.0.
392  Consider, e.g., a join between t1 and t2:
393 
394  "SELECT ... WHERE t1.col1=t2.colx and t2.coly OP @<something@>"
395 
396  where t1 is a prefix table and the optimizer currently calculates
397  the cost of adding t2 to the join. Assume that the chosen access
398  method on t2 is a 'ref' access on 'colx' that is estimated to
399  produce 2 rows per row from t1 (i.e., rows_fetched = 2). It will
400  in this case be perfectly fine to calculate a filtering effect
401  @<0.5 (resulting in "rows_fetched * filter_effect @< 1.0") from the
402  predicate "t2.coly OP @<something@>". If so, the number of row
403  combinations from (t1,t2) is lower than the prefix_rowcount of t1.
404 
405  The above is just an example of how the fanout of a table can
406  become less than one. It can happen for any access method.
407  */
408  float filter_effect;
409 
410  /**
411  prefix_rowcount and prefix_cost form a stack of partial join
412  order costs and output sizes
413 
414  prefix_rowcount: The number of row combinations that will be
415  joined to the next table in the join sequence.
416 
417  For a joined table it is calculated as
418  prefix_rowcount =
419  last_table.prefix_rowcount * rows_fetched * filter_effect
420 
421  @see filter_effect
422 
423  For a semijoined table it may be less than this formula due to
424  duplicate elimination.
425  */
426  double prefix_rowcount;
427  double prefix_cost;
